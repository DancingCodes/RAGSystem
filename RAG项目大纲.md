# RAG 项目大纲（Web 上传资料 + 向量库检索增强问答）

## 1. 项目目标

- 用户在 Web 端上传 PDF 资料，系统自动解析、切分、向量化并写入向量数据库。
- 用户在 Web 端提问时，系统基于向量库检索到的内容生成答案，并尽量提供引用来源（页码/段落/文件名）。
- 支持多用户/多知识库隔离，便于团队或个人分别管理资料。

## 2. 核心用户故事

- 作为用户，我可以注册/登录，并创建一个“知识库”。
- 作为用户，我可以上传一个或多个 PDF 文件，看到解析与入库进度。
- 作为用户，我可以在某个知识库下提问并获得带引用的回答。
- 作为用户，我可以查看检索命中的片段、来源文件与页码，并可追溯到原文。
- 作为用户，我可以删除文件/知识库，并同步清理向量索引。

## 3. MVP 范围（建议）

- Web：登录（可选）、知识库列表、文件上传页、问答页（含引用）。
- 文档：仅支持可复制文本的 PDF（扫描版 PDF 暂不支持）。
- 检索：向量检索 + 可选关键词混合检索；返回 TopK 片段。
- 生成：基于检索片段拼接上下文，输出答案 + 引用列表。
- 运维：基础日志、失败重试、简单的任务队列。

## 4. 技术选型

### 4.1 前端

- Next.js（React）
- TypeScript
- Tailwind CSS

### 4.2 后端

- Python FastAPI

### 4.3 存储（可选建议）

- 关系数据库：PostgreSQL（存用户、知识库、文件/任务/Chunk 元数据）
- 对象存储：本地磁盘（MVP）或 S3 兼容存储（生产）
- 向量数据库：根据部署形态选择（本地可选 Chroma；生产可选 Milvus/Qdrant/pgvector）

### 4.4 落地决策（当前）

- 大模型：DeepSeek（通过 OpenAI 兼容接口调用；密钥仅通过环境变量配置，不写入代码与文档）
- Embedding：接入 Embeddings + 向量数据库的真实 RAG（模型名与接口地址通过环境变量配置）
- 向量数据库：先本机部署（MVP 跑通），后续迁移到云上托管/集群
- 用户与权限：先单用户（不做登录），知识库隔离以 `knowledge_base_id` 为边界
- 文件存储：上传 PDF 先保存到项目本地目录（后续再切换对象存储）
- PDF 约束：先简化处理规则（仅支持可复制文本 PDF；扫描版暂不支持）
- 输出格式：先采用“答案 + 引用列表”（引用包含文件名、页码、命中文本片段）

### 4.5 配置与密钥（建议）

- 后端使用 `.env` 本地文件或系统环境变量配置：模型接口地址、API Key、模型名
- 不要把 API Key 提交到仓库；如不慎泄露，立即轮换 Key

## 5. 总体架构（逻辑模块）

### 5.1 Web 前端

- 文件上传：拖拽/选择文件、显示队列、进度与状态。
- 问答界面：输入问题、展示答案、引用片段与来源。
- 管理界面：知识库/文件列表、删除、重命名。

### 5.2 后端 API

- 认证与授权：用户 -> 知识库 -> 文件/片段权限。
- 上传与存储：接收文件，写入对象存储/本地存储，落库元数据。
- 任务编排：解析/切分/向量化/入库异步执行；失败重试。
- 问答接口：检索 -> 构造提示词 -> 调用模型 -> 返回答案与引用。

### 5.3 文档处理流水线（Ingestion Pipeline）

- 文档解析：PDF 文本抽取（仅支持可复制文本的 PDF；扫描版暂不支持）。
- 清洗与规范化：去页眉页脚、空白、重复换行，保留页码。
- 切分（Chunking）：按段落/标题/长度切分，配置重叠。
- 向量化（Embedding）：将 chunk 转成向量；保留元数据。
- 入库：写入向量库 + 关系库（元数据、状态、可追溯信息）。

### 5.4 存储层

- 关系数据库：用户、知识库、文件、chunk 元数据、任务状态、对话记录（可选）。
- 对象存储：原始文件、解析中间产物（可选）。
- 向量数据库：chunk 向量、元数据（file\_id、page、offset、标题等）。

### 5.5 AI 能力层

- 检索：向量检索 TopK；可选 rerank。
- 生成：基于检索上下文回答；要求“只依据提供材料回答”。
- 引用：输出引用列表（文件名、页码、chunk 摘要）。

## 6. 数据流（端到端）

### 6.1 上传入库

1. 前端上传文件 -> 后端保存文件与元数据
2. 触发异步任务：解析 -> 清洗 -> 切分 -> 向量化 -> 写入向量库
3. 更新任务状态：pending/running/succeeded/failed
4. 前端轮询或 WebSocket 获取进度

### 6.2 提问回答

1. 前端提交问题（指定 knowledge\_base\_id）
2. 后端对问题向量化，向量库检索 TopK chunk
3. （可选）对 TopK rerank/去重/合并相邻片段
4. 构造上下文与提示词，调用大模型生成答案
5. 返回：答案 + 引用（chunk 列表、来源、页码）+（可选）检索分数

## 7. 关键设计点（建议决策）

### 7.1 Chunk 设计

- 目标：保证语义完整、长度适配模型上下文、引用可追溯。
- 元数据建议：`user_id`、`knowledge_base_id`、`file_id`、`file_name`、`page_number`、`chunk_index`、`text_hash`、`created_at`。

### 7.2 检索策略

- 基础：向量 TopK。
- 提升：向量 + BM25 混合检索；或增加 rerank。
- 约束：按 `knowledge_base_id` 过滤，确保隔离。

### 7.3 提示词与防幻觉

- 系统指令：仅基于提供的资料回答；资料不足时明确说明“不确定/未找到”。
- 输出格式：答案 + 引用列表；引用必须来自检索到的 chunk。

### 7.4 多租户与权限

- 最小实现：所有查询/检索都必须带 `user_id`/`knowledge_base_id` 过滤。
- 审计：记录谁上传了什么、何时提问、命中哪些来源（可选）。

### 7.5 成本与性能

- 解析/向量化异步化，避免阻塞上传接口。
- 向量化缓存：对重复文本（`text_hash`）避免重复 embedding（可选）。
- 批处理 embedding：提升吞吐、降低单次开销。

## 8. API 草案（示例）

- `POST /api/knowledge-bases` 创建知识库
- `GET /api/knowledge-bases` 列表
- `POST /api/files` 上传 PDF 文件（关联 knowledge\_base\_id）
- `GET /api/files?knowledge_base_id=...` 文件列表与状态
- `GET /api/jobs/:id` 入库任务状态
- `POST /api/chat` 提问（knowledge\_base\_id + question）

## 9. 数据模型草案（示例）

- users：用户信息
- knowledge\_bases：知识库
- files：文件元数据、存储位置、解析状态
- chunks：chunk 文本、页码、位置、hash
- embeddings：向量记录（也可直接在向量库中承担）
- jobs：异步任务与日志
- chats/messages（可选）：对话记录

## 10. 质量与评测（建议）

- 可用性：上传成功率、入库成功率、平均入库耗时、检索耗时、回答耗时。
- 正确性：答案是否引用到正确来源；是否出现无依据内容。
- 回归集：用固定问题集与期望引用页码做自动化评测（可选）。

## 11. 里程碑（建议）

- M1：PDF 上传 + 入库任务 + 向量检索 + 基础问答（带引用）
- M2：权限隔离完善 + 文档列表/删除 + 失败重试与监控
- M3：混合检索/rerank + OCR + 更强的引用与可视化
